# Checkpoint 2 - Research Contract Adviser Agent

## Team 12 - NovaMind Collective

Our team is developing a proof-of-concept Research Contract Adviser Agent to support university staff in reviewing externally drafted research contracts. The system is intended to help reviewers identify the likely contract type, compare contract clauses against UoA reference templates and contracting positions, detect possible deviations, and generate structured Green / Amber / Red / Blue review flags. It is designed as a human-in-the-loop review support tool and does not provide legal advice, approve contracts, or replace professional contract reviewers.

Our solution follows a modular retrieval-augmented and rule-guided architecture. The current pipeline is: document input → text extraction → contract type recognition → clause segmentation → reference document loading → reference retrieval → clause comparison → flag generation → Markdown / JSON report generation → Streamlit demo UI. This design was selected because contract review requires traceable outputs grounded in trusted reference materials rather than unsupported free-form chatbot responses.

The current MVP has implemented the main end-to-end workflow. The Document Loader reads TXT, PDF, and DOCX files. The Contract Classifier predicts the likely contract type using weighted keyword and filename signals. The Clause Segmenter converts extracted text into structured clause objects. The Reference Loader loads UoA templates and contracting position documents from local folders. The Reference Retriever retrieves relevant reference chunks for each uploaded clause. The Clause Comparator assigns Green, Amber, Red, or Blue flags using retrieved evidence and rule-based risk indicators. The Report Generator produces Markdown and JSON review reports, and the Streamlit UI presents the full review process in a web interface.

The system currently uses local UoA reference materials from `data/reference_templates/`, `data/reference_positions/`, and `docs/use_case/`. These include UoA contract templates, UoA Contracting Positions and Approvals / Escalation Protocol, and the Research Contract Adviser Agent use case brief. These materials are stored locally for development and demonstration and should not be uploaded to the public GitHub repository.

The current demo supports two sample contracts. For the safe sample contract, the system identifies a Research Collaboration Agreement, detects 7 clauses, and produces 0 Red, 2 Amber, 0 Blue, and 5 Green flags. For the risky sample contract, the system identifies a Research Collaboration Agreement, detects 7 clauses, and produces 6 Red, 1 Amber, 0 Blue, and 0 Green flags. The risky sample demonstrates that the system can detect high-risk wording related to confidentiality, intellectual property assignment, publication restrictions, unlimited liability, termination restrictions, and non-New Zealand governing law.

The automated test suite currently passes with `8 passed in 4.17s`. The tests cover document loading, clause segmentation, contract type classification, reference document loading, reference retrieval, safe sample flag counts, risky sample red flag detection, and Markdown / JSON report generation. This provides evidence that the main MVP pipeline is functioning consistently.

The Streamlit UI has also been implemented and tested. It allows users to select a safe sample contract, select a risky sample contract, upload TXT / PDF / DOCX files, run the review pipeline, view document summary information, inspect Red / Amber / Blue / Green flag counts, review clause-level details, and download Markdown or JSON reports.

Key responsible AI considerations include confidentiality, explainability, and human oversight. The system does not provide legal advice, make final legal conclusions, or approve or reject contracts. Each output includes the clause text, retrieved reference source, review flag, rationale, and a human review requirement.

Current limitations include rule-based clause segmentation, lexical reference retrieval, rule-based flag generation, a small benchmark set, and no production Azure AI Search or Microsoft Foundry integration yet. Next steps include refining the UI for final demonstration, preparing presentation slides, improving architecture explanations, cleaning the repository before final GitHub upload, and ensuring internal UoA reference files are excluded from GitHub.
